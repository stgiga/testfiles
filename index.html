stgiga's Test Files

Welcome!

To begin, I will say that this page is designed to be a unified place for where test files I have made over the years in various technology sectors live. They can help you squash errors in whatever you do.

If any of these break, it means that there is something wrong with your infrastructure. It's important to note that more than a few of these test files ended up being useful as tests unintentionally.

Basically, what that means is that the relevant content of mine went so all-out that things go wrong when certain environments try to comprehend them, with varying degrees of severity.

And then you have files that are great benchmarks or litmus tests of quality and/or rarer features, or things that push the limits of what can be done, or seem outright impossible to do or encounter.

As for intentional tests, those DO exist, and are counted. Now that we have that out of the way, I should begin the list.

UnifontEX:

Features:

  • Support for 65,439 glyphs (more coming), covering many Unicode blocks
  • Consistent 16px pixel grid design for all characters
  • Improved renderer support, including IDEs
  • Extensive support for ancient scripts and modern emoji
  • Better feature utilization
  • MANY font formats, including obscure ones

Test File Rationale:

UnifontEX is a libre font fast approaching 2^16 glyphs, and this has confused quite a few programs, especially due to higher-Plane characters, including all sorts of technical and other symbols, including ones useful for debugging.

Test File Rationale:

This CC-BY-SA4(+GPL3) SoundFont is just under a prevalent artificial, unneeded size cutoff for SoundFonts in some programs, and by a tiny margin, yet it makes a GREAT Silicon SoundFont (Section 11 of the SoundFont standard).

8K UHD Galaxy Brain Wallpaper:

Test File Rationale:

This image is GREAT for testing image compression algorithms due to how vibrant, detailed, and large it is.

Part of it includes this 3D model which is a challenging model to 3D print, but can ONLY be made physical via it.

It also includes the Flashpoint Archive icon of this:

This is a carefully-GZipped SVG that in actuality is an HTML5 browser fidget toy AND a superset of a Chrome Experiment.

As you can imagine, this is ALSO a great test image, as is this:
This is an X-Face, X-Face-el X-Face-Xmas 8bpp grayscale X-Face, Face: header (incidentally using the rare PNG 2bpp Indexed Color mode for space reasons),
a Datula animated beyond-3bpp (12 bits but 24 is the format limit which I didn't remotely hit and wouldn't fit in Datula's custom header sending box) RGB animated X-Face,
and an X-Face-el X-Face-Xmas 96x48 ("multi") 3bpp RGB animated X-Face, all with my art.


As you can imagine, shoving ALL that into a .xface or .face file is something that wouldn't be expected but isn't against the rules given people really did send X-Face-Xmas X-Faces and Datula X-Faces around, and those use more than one X-Face header.
And of course inputting multiple headers manually would hurt. Ideally a decoder would do this grid layout given everything is 48x48 (generating the beyond-48x48 stuff is its own can of worms, but I did it).
Also, X-Face in an e-mail would and did GREATLY identify senders. Now that pixel is cool again, why shouldn't we bring them back?
